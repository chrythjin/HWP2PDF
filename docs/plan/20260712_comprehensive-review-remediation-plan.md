# HWP2PDF 종합 코드 리뷰 개선 계획

> 상태: 로컬 구현 및 검증 완료 (2026-07-12). GCP staging 검증과 기존 추적 산출물 삭제는 별도 운영 작업으로 남아 있다.

## 1. 목표

2026-07-12 종합 코드 리뷰에서 확인된 14개 항목을 운영 장애와 데이터 손실 위험 순으로 수정한다. 단순히 정적 검사와 단위 테스트를 통과하는 데 그치지 않고, 업로드, 큐 등록, 워커 변환, 상태 조회, 다운로드로 이어지는 실제 서비스 경계에서 동작을 검증한다.

## 2. 완료 기준

- 변환 실패가 작업 상태와 워커 HTTP 응답에 일관되게 반영되고, 재시도 가능한 오류는 Cloud Tasks 재시도를 유도한다.
- 큐 등록 실패가 성공 응답 뒤에 유실되지 않으며 작업이 영구 `queued` 상태로 고립되지 않는다.
- 비동기 워커는 업로드 요청을 처리한 Cloud Run 인스턴스의 로컬 파일에 의존하지 않는다.
- 직접 업로드 완료 시 GCS 실제 객체의 크기와 콘텐츠 타입을 검증한다.
- 게시글 PATCH는 요청에 포함된 모든 필드를 독립적으로 검증한다.
- 작업 삭제, 보존기간, 상태 전이와 게시글 갱신의 소유권 및 동시성 규칙이 저장소 구현에 강제된다.
- 프록시 IP, HWP 파일 시그니처, Firestore 인덱스, 문서 언어 및 임시파일 관리가 운영 계약과 일치한다.
- 관련 회귀 테스트, 타입 검사, 린트, 전체 빌드가 통과한다.
- 가능한 로컬 표면에서 API의 정상 입력과 비정상 입력을 직접 호출해 응답을 확인한다.

## 3. 구현 단계

### 단계 1. 변환 실행 경계와 작업 유실 방지

상태: 완료

대상:

- `apps/api/src/services/conversion-service.ts`
- `apps/api/src/services/cloud-tasks-dispatcher.ts`
- `apps/api/src/routes/v1.ts`
- 관련 API 테스트

작업:

1. 변환 서비스가 실패 상태를 기록한 뒤 오류를 호출자에게 다시 전달하도록 계약을 정리한다.
2. 워커가 재시도 가능 오류에는 비-2xx, 영구 오류에는 명시적 완료 응답을 반환하도록 상태와 HTTP 의미를 일치시킨다.
3. 업로드 완료 경로가 큐 등록 결과를 기다리고, 실패 시 작업 상태를 실패로 전환한 뒤 호출자에게 안정적인 오류를 반환한다.
4. GCS 원본 객체가 있는 작업은 워커 실행 인스턴스의 임시 경로로 내려받은 뒤 변환하고 즉시 정리한다.
5. 중복 Cloud Tasks 전달에 대한 기존 `queued` claim 원자성을 유지한다.

검증:

- 변환 성공, 영구 실패, 재시도 가능 실패 테스트
- 큐 등록 실패 시 작업 상태와 HTTP 응답 테스트
- 업로드 인스턴스의 로컬 파일 없이 GCS 원본만으로 워커가 처리되는 테스트
- 중복 워커 전달 회귀 테스트

### 단계 2. 외부 입력과 업로드 검증 강화

상태: 완료

대상:

- `apps/api/src/routes/v1.ts`
- `apps/api/src/routes/v1.board.ts`
- `apps/api/src/services/storage-service.ts`
- `packages/shared/src/index.ts` 또는 API 파일 검증 모듈
- 관련 테스트

작업:

1. GCS 객체 metadata 조회 경계를 추가하고 완료 요청의 신고값과 실제 크기·콘텐츠 타입·객체 경로를 비교한다.
2. 검증 실패 객체는 작업 생성과 큐 등록 전에 거부하고 필요 시 안전하게 정리한다.
3. PATCH 전용 검증으로 제공된 `title`, `body`, `category`를 각각 독립 검증한다.
4. 확장자뿐 아니라 HWP v5 OLE 시그니처를 최소 경계로 검사하되, 기존 테스트 fixture와 실제 지원 형식을 확인해 호환성을 보존한다.

검증:

- 실제 객체 크기 불일치, 콘텐츠 타입 불일치, 객체 없음 테스트
- 빈 본문, 과도한 길이, 잘못된 category를 포함한 PATCH 행렬 테스트
- 정상 HWP 시그니처와 위장 파일 테스트

### 단계 3. 보존·소유권·동시성 일관성 확보

상태: 로컬 구현 완료. Firestore 실제 staging 쿼리 검증은 미수행.

대상:

- `packages/shared/src/index.ts`
- `apps/api/src/services/job-store.ts`
- `apps/api/src/services/board-store.ts`
- `apps/api/src/routes/v1.ts`
- Firestore 설정과 관련 테스트

작업:

1. 익명 작업, 회원 다운로드, 회원 메타데이터의 만료 시점을 명시적으로 분리한다.
2. tombstone에 소유권 검증에 필요한 최소 정보를 보존하고 비소유자에게는 일관된 `404`를 반환한다.
3. Firestore 작업 갱신과 삭제를 transaction 및 허용된 상태 전이 규칙으로 보호한다.
4. 게시글 수정과 삭제를 transaction 또는 버전 기반 조건부 갱신으로 보호한다.
5. 회원 이력 쿼리에 필요한 Firestore 복합 인덱스를 저장소에서 관리한다.

검증:

- 익명/회원 만료 필드 및 조회 동작 테스트
- 소유자/비소유자/삭제 후 반복 요청 권한 행렬 테스트
- 갱신 대 삭제, 중복 워커, 게시글 동시 수정 충돌 테스트
- Firestore emulator 또는 실제 staging에서 인덱스 쿼리 검증

### 단계 4. 운영 경계·접근성·저장소 위생

상태: 코드 및 ignore 규칙 완료. 기존 추적 임시파일과 `package-lock.json` 삭제는 승인 없이 수행하지 않음.

대상:

- `apps/api/src/app.ts`
- `apps/web/src/app/layout.tsx`
- `.gitignore`
- 배포·운영 문서

작업:

1. Cloud Run의 신뢰 가능한 프록시 홉을 Express 설정으로 명시하고 rate limit 키를 검증된 `request.ip`에서 생성한다.
2. 변환 프로세스의 기존 시간 제한과 임시 디렉터리 격리를 재확인하고 파일 검증 계약을 운영 문서에 기록한다.
3. HTML 문서 언어를 `ko`로 수정한다.
4. API 임시 업로드 경로가 추적되지 않도록 유지하고, 추적 중인 런타임 산출물과 불필요한 npm lockfile은 사용자 변경과 의도를 확인한 뒤 별도 정리한다.

검증:

- 직접 IP와 전달 헤더 조합별 rate limit key 테스트
- 접근성 정적 검사와 브라우저 문서 언어 확인
- `git status` 및 추적 파일 목록 확인

## 4. 검증 순서

1. 변경 파일 LSP 진단 또는 TypeScript 컴파일러 진단
2. 변경 동작별 표적 Vitest
3. `pnpm typecheck`
4. `pnpm -r test`
5. `pnpm -r --if-present lint`
6. `pnpm -r build`
7. `ocr review`
8. 로컬 API 정상·오류 흐름 수동 호출
9. GCP staging에서 Cloud Tasks, GCS, Firestore 인덱스 및 인스턴스 경계 검증

## 5. 위험 관리

- 현재 작업 트리의 사용자 변경은 수정하거나 되돌리지 않는다.
- Firestore 문서 형태 변경은 기존 지속 데이터와 호환되도록 점진적으로 읽을 수 있게 설계한다.
- 운영 리소스 변경, 배포, 추적 파일 삭제는 별도 승인 없이 실행하지 않는다.
- 로컬에서 재현할 수 없는 GCP 동작은 자동 테스트와 mock 결과를 실환경 검증으로 표현하지 않고 staging 미검증 항목으로 명시한다.

## 6. 완료 결과

- 변환 오류를 호출자까지 전파하고 재시도 가능 오류는 작업을 `queued`로 되돌린 뒤 워커가 500을 반환하도록 수정했다.
- 큐 등록을 await하고 실패 작업을 `failed`로 기록하며, 직접 업로드 세션도 실패 시 종결한다.
- GCS 기반 워커가 원본을 자신의 임시 경로로 내려받고 최초 신고 크기를 재검증하도록 원본 크기를 작업 레코드에 보존한다.
- 직접 업로드의 실제 GCS metadata와 HWP OLE 시그니처를 검증하고 실패 파일을 정리한다.
- 게시글 PATCH 필드별 검증, 작업 및 게시글 Firestore transaction, tombstone 최소 소유권 보존을 적용했다.
- 회원 다운로드와 메타데이터 보존기간을 분리하고, Firestore 회원 이력 쿼리의 부등식 제약을 제거했다.
- Express 프록시 신뢰 설정과 `request.ip` 기반 rate limit 키, 한국어 문서 언어 선언, Firestore 인덱스 선언을 적용했다.
- 전체 404개 테스트, 전체 프로덕션 빌드, 웹 린트, 변경 파일 진단과 실제 로컬 HTTP 경계를 검증했다.
- GCP Cloud Tasks/GCS/Firestore 인스턴스 경계 및 인덱스 배포는 로컬 결과로 대체하지 않았으며 staging 검증이 남아 있다.
