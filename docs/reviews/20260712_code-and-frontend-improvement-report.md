# HWP2PDF 코드·프론트엔드 개선 검토 보고서

- 검토일: 2026-07-12
- 범위: `apps/api`, `apps/web`, `packages/shared`, 배포·Firestore 설정

- 목적: 이미 해결된 결함을 제외하고 다음 개선 주기의 우선순위를 제시한다.

## 요약

현재 구현은 핵심 업로드·변환·다운로드·인증·게시판 흐름과 주요 보안 경계가 갖춰져 있다. 다음 개선 주기에서는 새 기능보다 **장애 복구 가능성**, **오류 계약의 안전성**, **사용자에게 정확한 상태 전달**, **모바일·보조기술 접근성**을 먼저 강화하는 편이 효과가 크다.

확인된 개선 후보는 총 12건이다.

## P0: 즉시 개선 권장

### 1. stuck 작업과 만료 업로드 세션을 실제로 복구할 실행 경로가 없다

**근거**

- `apps/api/src/services/cloud-tasks-dispatcher.ts:324`에 `isJobStuck()`이 있다.
- `apps/api/src/services/cloud-tasks-dispatcher.ts:345`에 `resetStuckJob()`이 있으나 운영 스케줄러나 관리 엔드포인트에서 호출되지 않는다.
- `apps/api/src/services/job-store.ts:186`의 `expireUploadSessions()`도 구현만 있고 실행 주체가 없다.

프로세스 충돌, Cloud Tasks 전달 소진, 브라우저 이탈이 발생하면 `processing` 작업 또는 미완료 업로드 세션이 계속 남을 수 있다. 복구 함수가 존재해도 호출·관측·재등록 경로가 없으면 운영 복구 기능으로 볼 수 없다.

**권장 변경**

1. 인증된 내부 maintenance 엔드포인트를 추가한다.
2. Cloud Scheduler가 주기적으로 해당 엔드포인트를 호출하게 한다.
3. stuck 작업은 transaction으로 현재 상태와 `updatedAt`을 재검증한 뒤 `queued`로 복원하고 Cloud Tasks에 다시 등록한다.
4. 만료 업로드 세션은 상태를 종결하고 고아 GCS 원본을 삭제한다.
5. 처리 건수, 실패 건수, 가장 오래된 작업 연령을 구조화 로그로 남긴다.

**완료 기준**

- 장애 주입으로 `processing`에 남긴 작업이 한 번만 재등록된다.
- 만료 세션과 고아 객체가 정리된다.
- maintenance 호출 중 정상 변환 작업과 경합해도 상태가 되돌아가지 않는다.

### 2. 변환 엔진의 원본 오류 메시지가 사용자 응답에 노출될 수 있다

**근거**

- `apps/api/src/services/conversion-service.ts:55`가 잡은 예외의 `error.message`를 작업 `message`에 그대로 저장한다.
- 작업 상태 응답은 이 메시지를 프론트엔드 실패 화면에 표시한다.

LibreOffice 오류에는 로컬 경로, 명령 인자, 확장 기능 또는 런타임 환경 정보가 포함될 수 있다. 이는 사용자에게 유용하지 않은 내부 정보이면서 환경 노출 가능성이 있다.

**권장 변경**

- 저장·응답용 공개 오류 코드와 내부 진단 메시지를 분리한다.
- 사용자에게는 `invalid_document`, `conversion_timeout`, `converter_unavailable`, `conversion_failed`처럼 제한된 코드와 한국어 메시지만 제공한다.
- 원본 예외는 `requestId`, `jobId`, 오류 분류와 함께 서버 로그에만 남긴다.

**완료 기준**

- LibreOffice가 경로를 포함한 오류를 내도 API 응답과 Firestore 공개 필드에 경로가 나타나지 않는다.
- 운영 로그에서는 동일 사건을 `requestId` 또는 `jobId`로 추적할 수 있다.

### 3. 이력 조회 장애가 “변환 이력이 없음”으로 오인될 수 있다

**근거**

- 이력 페이지는 조회 실패 시 빈 배열로 대체하는 흐름을 사용한다.
- `apps/web/src/components/JobHistoryList.tsx:69`는 빈 배열이면 무조건 “변환 이력이 없습니다.”를 표시한다.

네트워크·인증·서버 장애와 실제 빈 이력을 구분하지 않으면 사용자는 데이터가 삭제됐다고 오해할 수 있고 재시도 방법도 알 수 없다.

**권장 변경**

- 상태를 `loading | success | empty | error`로 명시적으로 분리한다.
- 오류 화면에 재시도 버튼과 안전한 오류 설명을 제공한다.
- 기존 목록이 있던 상태에서 갱신이 실패하면 목록을 유지하고 비차단 경고를 표시한다.

**완료 기준**

- API 500·네트워크 중단·401 각각이 빈 이력과 다른 화면을 보인다.
- 재시도로 정상 목록을 복구할 수 있다.

### 4. “변환 완료”와 실제 다운로드 가능 상태가 일치하지 않는다

**근거**

- `apps/web/src/components/DropzoneUploader.tsx:504`는 `status === "completed"`이면 항상 “PDF 파일이 준비되었습니다.”와 다운로드 버튼을 표시한다.
- `apps/web/src/components/JobHistoryList.tsx:27`은 `downloadExpiresAt`이 없는 완료 작업을 만료되지 않은 것으로 판단한다.

완료 상태여도 결과 객체 누락, URL 발급 실패, 다운로드 만료가 발생할 수 있다. 현재 UI는 사용자가 버튼을 누른 뒤에야 모순을 알게 된다.

**권장 변경**

- API 계약에 `downloadAvailable` 또는 결과 객체 존재·만료를 반영한 명시적 필드를 추가한다.
- 완료 화면은 `completed + downloadable`, `completed + expired`, `completed + unavailable`을 분리한다.
- `downloadExpiresAt`이 없는 완료 작업은 보수적으로 “확인 필요”로 처리한다.

**완료 기준**

- 결과 삭제, URL 발급 실패, 만료된 작업에서 활성 다운로드 버튼이 표시되지 않는다.
- 사용자에게 변환 자체 성공과 다운로드 불가 사유가 구분되어 표시된다.

## P1: 다음 개발 주기 권장

### 5. 게시판 카테고리 조회용 Firestore 인덱스를 저장소에서 관리해야 한다

게시판 목록은 카테고리 필터와 생성일 정렬을 결합한다. 현재 저장소의 `firestore.indexes.json`에는 작업 이력 인덱스만 관리되므로 실제 게시글 수가 늘고 필터 쿼리가 실행될 때 운영에서 인덱스 오류가 발생할 수 있다.

**권장 변경:** `boardPosts(category ASC, createdAt DESC)` 복합 인덱스를 선언하고 staging에서 각 카테고리 쿼리를 실행해 `READY` 상태와 결과를 확인한다.

### 6. 변환 실행 책임을 API 프로세스에서 분리해야 한다

현재 HTTP API, Cloud Tasks worker, GCS 다운로드, LibreOffice 프로세스 실행이 같은 애플리케이션과 컨테이너에 결합돼 있다. 이 때문에 변환기만 네트워크를 차단하거나 별도 CPU·메모리 정책을 적용할 수 없다.

**권장 변경:** API는 업로드·인가·작업 상태만 담당하고, 별도 private Cloud Run converter 서비스가 GCS 원본을 받아 변환하도록 분리한다. converter 서비스에는 전용 서비스 계정, 최소 GCS 권한, 동시성 1, 제한된 egress와 강한 자원 제한을 적용한다.

### 7. 모바일에서 주요 회원 내비게이션이 사라진다

`apps/web/src/components/PageLayout.tsx:37`의 이력·게시판 내비게이션은 `hidden md:flex`이고, `AuthNav`에는 대체 링크가 없다. 작은 화면에서 로그인 사용자는 홈에서 이력이나 게시판으로 이동할 명시적 경로가 없다.

**권장 변경:** 모바일 메뉴 버튼과 키보드 접근 가능한 내비게이션 패널을 추가하거나, 작은 화면에서도 핵심 링크를 유지하는 압축형 내비게이션을 제공한다.

### 8. 업로드·변환 진행 상태를 보조기술에 전달해야 한다

`DropzoneUploader`의 진행 막대는 시각적 `div`이며 `role="progressbar"`, `aria-valuenow`, 상태 변화용 `aria-live`가 없다. 실패 메시지도 다운로드 오류를 제외하면 일관된 live region이 아니다.

**권장 변경:** 진행 막대에 progressbar semantics를 추가하고, 업로드·대기·변환·완료·실패 전환을 `aria-live="polite"` 또는 중요한 실패의 `role="alert"`로 전달한다. 장식용 SVG에는 `aria-hidden="true"`를 적용한다.

### 9. 게시판 필터와 페이지 이동의 선택 상태·문맥이 부족하다

`apps/web/src/app/board/page.tsx:137`의 카테고리 버튼은 선택 상태를 색상으로만 표현하며 `aria-pressed`가 없다. 페이지 번호 영역도 navigation landmark와 접근 가능한 이름이 없다.

**권장 변경:** 카테고리 버튼에 `aria-pressed`, 필터 그룹 이름을 추가하고 페이지 이동 영역을 `<nav aria-label="게시판 페이지">`로 구성한다. 목록 갱신 중 기존 목록을 유지하고 해당 영역에 `aria-busy`를 사용한다.

## P2: 품질 개선

### 10. 삭제 확인을 네이티브 `window.confirm`에서 애플리케이션 대화상자로 전환

게시글과 이력 삭제가 `window.confirm`에 의존한다. 브라우저별 표현 차이가 크고, 처리 중 상태·오류·대상 정보를 같은 문맥에서 제공하기 어렵다.

`<dialog>` 또는 접근 가능한 모달을 사용해 삭제 대상, 비가역성, 취소·확인 버튼과 진행 상태를 제공하는 것이 좋다. 모달 도입 시 포커스 이동·복귀, Escape, 배경 상호작용 차단을 함께 검증해야 한다.

### 11. 이력의 주요 식별 정보를 작업 ID에서 파일명으로 변경

`apps/web/src/components/JobHistoryList.tsx:94`는 긴 `jobId`를 주요 제목으로 표시한다. 사용자는 자신이 변환한 파일을 작업 ID보다 파일명으로 기억한다.

원본 파일명을 주요 제목으로, 작업 ID는 복사 가능한 보조 정보로 표시하고 실패 상태에서는 안전한 오류 요약을 함께 제공하는 것이 좋다.

### 12. 클라이언트 데이터 요청과 오류 처리를 공통화

게시판·이력 페이지가 `loading`, `error`, 인증 상태, 재요청 로직을 각각 구현한다. 이 구조는 오류를 빈 상태로 바꾸거나 페이지마다 401/403 처리 문구가 달라지는 문제를 만들기 쉽다.

프로젝트 규모를 고려해 대형 상태관리 도구를 도입하기보다, 우선 `fetchWithAuth` 위에 타입화된 응답 파서와 공통 오류 타입을 두고 페이지별 소형 hook으로 `loading/success/empty/error` 상태를 통일하는 것이 적절하다.

## 권장 실행 순서

### 1단계: 상태·오류 정확성

1. 공개 변환 오류 코드 도입
2. 이력 조회 상태 분리와 재시도
3. 다운로드 가능성 API 계약 및 UI 상태 분리

### 2단계: 운영 복구

1. maintenance 서비스 계약 정의
2. stuck 작업 transaction 복구 및 재등록
3. 만료 세션·고아 객체 정리
4. Cloud Scheduler와 구조화 관측 지표 연결

### 3단계: 프론트엔드 접근성·모바일

1. 모바일 내비게이션
2. 업로드 progress/live region
3. 게시판 필터·pagination semantics
4. 접근 가능한 삭제 대화상자

### 4단계: 운영 구조

1. 게시판 인덱스 선언·배포
2. converter 서비스 분리 설계
3. staging 부하·장애 주입 후 점진적 전환

## 검증 전략

- API: 상태 전이·오류 코드·maintenance 경합을 Vitest와 Firestore emulator 또는 staging으로 검증한다.
- 프론트엔드: React Testing Library로 `loading/error/empty/success`와 키보드 상호작용을 고정한다.
- 접근성: axe 자동 검사와 실제 키보드·스크린리더 수동 검증을 병행한다.
- 운영: Cloud Tasks 재시도 소진, worker 강제 종료, GCS 결과 누락, Firestore 일시 오류를 staging에서 장애 주입한다.
- 사용자 표면: 모바일 viewport에서 로그인 → 이력 → 다운로드 오류 복구 → 게시판 필터·페이지 이동을 브라우저로 직접 수행한다.

## 이번 보고서에서 제외한 항목

- 2026-07-12 종합 개선에서 이미 해결한 큐 등록 대기, GCS 원본 재다운로드, 직접 업로드 크기·OLE 검증, Firestore transaction, 프록시 IP, 회원 보존기간, 한국어 문서 언어 선언은 중복 기재하지 않았다.
- `timeout.unref()`는 자식 프로세스 핸들이 별도로 이벤트 루프를 유지하므로 현재 코드만으로 조기 프로세스 종료 결함이라고 확정하지 않았다.
- CSP는 AdSense·Firebase·GCS signed URL 등 실제 허용 대상 목록과 배포 응답 헤더를 함께 설계해야 하므로, 단편적인 헤더 추가 권고 대신 별도 보안 하드닝 작업으로 다루는 것이 적절하다.
- 색상·간격·애니메이션 같은 미관 선호는 사용자 행동이나 접근성에 직접 영향을 주는 근거가 없는 한 개선 목록에서 제외했다.
